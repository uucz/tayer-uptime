Moyu: An Empirical Study on Anti-Over-Engineering

Quantifying the Impact of Structured Prompt Strategies on AI Code Over-Engineering

摘要

大型语言模型 (LLM) 在代码生成任务中系统性地表现出过度工程倾向——添加未请求的注释、异常处理、类型检查和抽象层。 本研究提出 Moyu（摸鱼），一种结构化的反过度工程 Prompt 策略，并通过跨 3 个主流模型（Claude Sonnet 4、GPT-4o、Gemini 2.5 Pro）、 5 种实验条件（含消融对照）、12 个编程场景的 540 次受控实验进行了严谨评估。

结果表明，Moyu-Standard 相比无干预基线，在 A 类场景（过度工程高发区）中将代码输出行数减少约 —%， 过度工程信号综合得分下降 —%，同时保持 —% 的功能正确率。 关键消融发现：简单的 "Be concise" 指令仅能实现 Moyu 效果的一小部分，证实了结构化规则体系的不可替代价值。

Abstract

Large Language Models (LLMs) systematically exhibit over-engineering tendencies in code generation—adding unrequested comments, exception handling, type checks, and abstraction layers. This study presents Moyu, a structured anti-over-engineering prompt strategy, rigorously evaluated through 540 controlled experiments across 3 major models (Claude Sonnet 4, GPT-4o, Gemini 2.5 Pro), 5 experimental conditions (including ablation controls), and 12 coding scenarios.

Results show that Moyu-Standard reduces code output by approximately —% in A-type scenarios (over-engineering hotspots), decreases the composite over-engineering score by —%, while maintaining —% functional correctness. A critical ablation finding: a simple "Be concise" instruction achieves only a fraction of Moyu's effect, confirming the irreplaceable value of structured rule systems.

背景与动机

2.1 AI 过度工程问题

当开发者要求 AI 编码助手"修复一个空指针 bug"时，典型的 LLM 输出不仅修复了 bug，还会：

  • 给所有函数添加 docstring
  • 引入 try-except 异常处理
  • 添加 isinstance 类型检查
  • 重构不相关的函数
  • 创建新的工具类和辅助文件

2.2 为什么这很重要

过度工程的 AI 输出增加 code review 负担、引入潜在回归风险、降低代码可维护性。在大规模团队中，这直接影响开发效率。

2.3 现有研究

研究表明，LLM 的过度工程倾向源于多个因素：RLHF 长度偏差（Saito 2023 — 奖励模型系统性偏好更长回答）、 讨好型人格（Anthropic ICLR 2024 — 模型倾向于过度满足用户期望）、 以及设计选择（修复"偷懒"问题后模型被推向"做更多"方向）。

2.4 研究空白

尽管过度工程问题被广泛讨论，目前缺乏针对反过度工程 Prompt 策略的严谨量化研究。现有工作多为经验报告或单模型案例分析。本研究填补了这一空白。

Background & Motivation

2.1 The AI Over-Engineering Problem

When developers ask AI coding assistants to "fix a null pointer bug," typical LLM output not only fixes the bug but also:

  • Adds docstrings to all functions
  • Introduces try-except error handling
  • Adds isinstance type checks
  • Refactors unrelated functions
  • Creates new utility classes and helper files

2.2 Why This Matters

Over-engineered AI output increases code review burden, introduces regression risks, and reduces maintainability. At scale, this directly impacts development velocity.

2.3 Prior Work

Research shows LLM over-engineering stems from multiple factors: RLHF length bias (Saito 2023 — reward models systematically prefer longer responses), sycophancy (Anthropic ICLR 2024 — models tend to over-satisfy user expectations), and design choices (fixing "laziness" pushed models toward "do more").

2.4 Research Gap

Despite widespread discussion, there is no rigorous quantitative study on anti-over-engineering prompt strategies. Existing work consists of anecdotal reports or single-model case studies. This research fills that gap.

3.1 Test Codebase

We use an intentionally flawed Python task manager (~110 lines app.py + 37 lines helpers.py), featuring: manual file operations (no context managers), null pointer bugs, missing return values, etc.

import json
import os
from datetime import datetime
from helpers import calculate_priority_score, format_task_display

DATA_FILE = "tasks.json"

def load_tasks():
    if not os.path.exists(DATA_FILE):
        return []
    f = open(DATA_FILE, "r")
    data = f.read()
    f.close()
    if not data:
        return []
    return json.loads(data)

def save_tasks(tasks):
    f = open(DATA_FILE, "w")
    f.write(json.dumps(tasks, indent=2))
    f.close()

def add_task(title, assignee=None, priority="medium"):
    tasks = load_tasks()
    task = {
        "id": len(tasks) + 1,
        "title": title,
        "status": "pending",
        "assignee": assignee,
        "priority": priority,
        "created_at": datetime.now().isoformat(),
    }
    tasks.append(task)
    save_tasks(tasks)
    return task

def complete_task(task_id):
    tasks = load_tasks()
    for task in tasks:
        if task["id"] == task_id:
            task["status"] = "completed"
            task["completed_at"] = datetime.now().isoformat()
            save_tasks(tasks)
            return task
    # BUG: when task not found, still tries to access task variable
    task["status"] = "completed"
    save_tasks(tasks)
    return task

def delete_task(task_id):
    tasks = load_tasks()
    new_tasks = []
    for task in tasks:
        if task["id"] != task_id:
            new_tasks.append(task)
    save_tasks(new_tasks)
    # BUG: doesn't return whether deletion actually happened

def list_tasks(status=None):
    tasks = load_tasks()
    if status:
        result = []
        for task in tasks:
            if task["status"] == status:
                result.append(task)
        return result
    return tasks

def search(query):
    tasks = load_tasks()
    results = []
    for task in tasks:
        if query.lower() in task["title"].lower():
            results.append(task)
    return results

Ablation Study

5.1 "Be concise" vs Moyu

简单的一句"Be concise"是否能达到 Moyu 的效果？答案是否定的。

Baseline-Concise 在 LOC 上仅实现了 Control → Moyu-Standard 缩减量的一小部分。结构化的规则体系、检测层和文化包提供了简单指令无法替代的精确行为控制。

5.2 Lite vs Standard

Moyu-Lite（仅三铁律，75 行）相比 Moyu-Standard（303 行）效果如何？Lite 版已能实现可观的缩减，但 Standard 版的检测层（分级系统）和文化包进一步提升了效果。

5.3 Standard vs Strict

更严格是否更好？Moyu-Strict 施加了更强的约束（20 行上限），但可能导致模型过度压缩，在部分场景中反而降低了代码质量。

5.4 B 类结果

在 B 类场景（用户显式请求重构、文档、测试）中，Moyu 正确地不抑制这些工作，各条件间的 LOC 差异无统计学显著性。

5.1 "Be concise" vs Moyu

Can a simple "Be concise" achieve Moyu's effect? The answer is no.

Baseline-Concise achieves only a fraction of the Control → Moyu-Standard reduction. Structured rules, detection layers, and culture packs provide precise behavioral control that simple instructions cannot replicate.

5.2 Lite vs Standard

How does Moyu-Lite (three core rules only, 75 lines) compare to Moyu-Standard (303 lines)? Lite already achieves notable reduction, but Standard's detection layer (graduated system) and culture pack further improve results.

5.3 Standard vs Strict

Is stricter better? Moyu-Strict imposes stronger constraints (20-line cap), but may cause over-compression, actually reducing code quality in some scenarios.

5.4 B-Type Results

In B-type scenarios (explicit refactoring, documentation, testing requests), Moyu correctly does not suppress this work — LOC differences across conditions are not statistically significant.

讨论

6.1 为什么 Moyu 有效

Moyu 的有效性来自三个互补机制：

  1. 身份锚定：将"Staff Engineer"的克制品质嵌入模型的角色认知
  2. 决策点干预：在模型即将过度工程的关键时刻提供具体约束
  3. 正面激励重定向：利用讨好型偏差，让模型认为"做少 = 被认可"

6.2 局限性

  • 单语言：仅测试了 Python，其他语言可能有不同表现
  • 单代码库：一个约 150 行的小型代码库，大型项目中效果可能不同
  • 3 个模型：虽覆盖三大 provider，但未测试开源模型
  • Temperature 固定：仅使用 0.7，不同 temperature 可能影响结果

Discussion

6.1 Why Moyu Works

Moyu's effectiveness stems from three complementary mechanisms:

  1. Identity anchoring: Embedding the restraint qualities of a "Staff Engineer" into the model's role perception
  2. Decision-point intervention: Providing specific constraints at critical moments when the model is about to over-engineer
  3. Positive incentive redirection: Leveraging sycophancy bias to make the model believe "doing less = being approved"

6.2 Limitations

  • Single language: Only Python tested; other languages may differ
  • Single codebase: A small ~150-line codebase; larger projects may show different effects
  • 3 models: Covers three major providers but excludes open-source models
  • Fixed temperature: Only 0.7 used; different temperatures may affect results

结论与启示

对 Prompt 工程从业者

简单的"be concise"远不够。结构化的反过度工程策略——包含身份锚定、具体行为约束、分级检测——能实现显著更好的效果。

对 AI 工具开发者

默认行为不应是"尽可能多做"。提供内置的范围控制机制，让用户能轻松限制 AI 的修改范围。

对使用 AI 的开发者

在你的 AI 工作流中加入 Moyu 类策略。不是让 AI 变懒，而是让 AI 像经验丰富的 Staff Engineer 一样知道什么不该做。

Conclusion & Implications

For Prompt Engineers

A simple "be concise" is far from sufficient. Structured anti-over-engineering strategies — with identity anchoring, specific behavioral constraints, graduated detection — achieve significantly better results.

For AI Tool Developers

Default behavior should not be "do as much as possible." Provide built-in scope control mechanisms that let users easily limit AI modification scope.

For Developers Using AI

Incorporate Moyu-style strategies in your AI workflow. Not to make AI lazy, but to teach it — like an experienced Staff Engineer — what not to do.

Future Work

  • Extend to multiple languages (TypeScript, Go, Rust)
  • Validate in real projects (>10k LOC)
  • Test open-source models (Llama, Qwen)
  • Longitudinal study: effect changes with long-term Moyu usage
